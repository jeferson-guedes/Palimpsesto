Palimpsesto

Project memory for coding agents that never erases the why.

A palimpsest is a manuscript written over an erased one, where the earlier text still shows through. The current understanding is rewritten in place — the reasoning behind it stays readable underneath.

View on GitHub → Quickstart

The problem

You spend a session settling something real — why Markdown over SQLite, why option B is out, what done means for this version. The reasoning is good, the call is made.

Then the session ends. Open a fresh one tomorrow, switch machines, move from one agent to another — and it asks the same questions again. The model isn't weak. It just never had anywhere durable to put the conclusion. The work survives as code; the why evaporates into a chat log nobody reopens.

Palimpsesto gives a project a durable memory: plain Markdown that holds the truth, and a local engine — ChromaDB + a synaptic graph — that recalls it. On-device embeddings, no API calls, no vendor: it runs on your machine and lives in your repo. The writing is convention-first (no CLI to hand-edit memory); the recall is powered by the engine.

How a memory is shaped

Two layers per page

A memory that evolves — a decision, an ongoing project, a piece of feedback — carries two layers. One keeps you moving; the other lets you trace your way back.

compiled_truth — rewritable

The current best understanding. Rewrite it as a whole when things improve.

timeline — append-only

How the understanding got here. You add to it; you never rewrite it.

## compiled_truth
Config is stored as Markdown files in the repo — diff-able in a PR,
human-readable, no migration story. Parsed at boot; no DB dependency.

## timeline
- 2026-01-12 · hypothesis · leaned toward SQLite (atomic writes, a query API)
- 2026-01-19 · reversal   · dropped it — config is small and read-mostly, so
                            migrations cost more than the query layer is worth
- 2026-01-19 · decision   · store config as Markdown, parsed at boot

The one rule

No silent edits

The most common way a knowledge base rots is the silent edit: someone changes the conclusion and the reasoning behind it disappears. When an understanding changes, you don't quietly overwrite it. You:

  1. Rewrite compiled_truth to the new state.
  2. Append a reversal line to timeline saying what changed and why.

Those two happen together, every time. The change of mind stays on the record, and the path back is traceable. That single habit is the whole point — it is why the reasoning is never lost.

Two layers, one tool

Files hold the truth. The engine makes it memory.

Neither half is an add-on. The files are the source of truth; the semantic engine is what turns a folder of Markdown into recall — it answers "didn't we discuss this before?" even when you don't know which file it lives in. Without the engine you have notes; with it you have memory.

the truth

Durable files

What is known

  • Curated Markdown, compiled_truth + timeline
  • Index read every session
  • Diffable in a PR, readable by a human or grep
  • Written by convention — no CLI needed
the recall

Semantic engine

How it's found again

  • Local ChromaDB + on-device embeddings — zero API cost
  • Similarity search, on demand
  • A Hebbian synaptic graph: co-retrieved memories wire together
  • Spreading activation drags in relevant neighbors

It all runs on your machine — nothing leaves it, no API calls, no vendor. The truth is plain files in git; the engine is local. The memory belongs to you, not to a model.

Quickstart

Clone and wire it in

setup.sh is pure Bash — zero dependencies. It links the skill into your agents and, optionally, scaffolds a memory directory.

git clone git@github.com:jeferson-guedes/Palimpsesto.git
cd Palimpsesto

./setup.sh                       # wire the skill into your agents (~/.claude, ~/.codex)
./setup.sh ~/my-project/memory   # also scaffold a memory dir for a project
./setup.sh --semantic            # also set up the optional semantic layer

From there it's just Markdown. Point your agent at PALIMPSESTO.md — the contract — and it learns the rest.

Not this

A rules file (CLAUDE.md / AGENTS.md). Great for standing instructions. Palimpsesto is structured, addressable knowledge with an audit trail — a different job. Use both.

A hosted memory service. A vendor's vector store or a model's built-in memory is a runtime you don't own. Palimpsesto's engine runs locally and its truth is plain files in git — the memory belongs to you, not to a model or a vendor.

Chat history. History is the process. A memory is the conclusion — the handful of judgments you'll need again.
